**DESIGN RATIONALE:**

1. **Component Inventory Verification:** All component IDs have been cross-referenced with the provided `DESIGN_SYSTEM_DATA` and verified for existence and type.

2. **Layout Decisions:** The layout uses a vertical structure, placing a header at the top for clarity and a scrollable list of notification settings below.  Each setting is a list item with a leading label, switch component for toggling, and trailing padding for visual consistency. 16px padding is used consistently for content sections for readability.

3. **User Flow Reasoning:** The user interacts directly with toggle switches to manage their preferences. No explicit save button is needed, as each toggle change is assumed to be saved immediately.

4. **Component Choices:** The `10:9912` (switch) component was chosen for its clear on/off state and familiar interaction pattern.  Alternatives like custom shapes or checkbox groups would not provide the same intuitive user experience in this context.

5. **Component Gaps:** The design system does not include a dedicated notification settings screen, but the provided components are sufficient. There was no need for custom native elements to overcome functional limitations.

6. **Visual Hierarchy:** A clear vertical layout establishes hierarchy.  Section headers (if used) would have larger text.  Each notification type setting is displayed as a distinct list item, improving visual clarity.

7. **Variant Selection Rationale:** For the switch component (`10:9912`), the "enabled" state is used by default. The `Selected` variant will dynamically change to "true" or "false" depending on user interaction.  Other variants (focused, hovered, pressed) are handled by default component behavior.

8. **Native Element Justification:** No native elements were needed, as the existing components adequately meet the functional needs of the design.
