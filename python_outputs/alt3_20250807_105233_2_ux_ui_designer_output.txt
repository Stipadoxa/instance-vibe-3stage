**DESIGN RATIONALE:**

1. **Component Inventory Verification:** I have verified all component references against the provided design system data.  The `10:9912` (switch) component is crucial for this design.

2. **Layout Decisions:** The page uses a vertical layout for a clear top-to-bottom flow of notification categories. Each category is presented as a list item (`10:10214`), providing a consistent structure.  Spacing is maintained using the standard 16px vertical and 16px horizontal padding defined in Pattern A.

3. **User Flow Reasoning:** Users will scan the list of notification types, and easily toggle each one on or off using the switch component.  The clear labeling and visual feedback from the switch provides immediate confirmation of their choices.

4. **Component Choices:** I chose the `10:10214` (List item) component to structure the notification settings because it naturally supports leading and trailing elements (the description text and the switch, respectively).

5. **Component Gaps:** The design system doesn't offer a built-in notification preference persistence mechanism.  This is handled outside of this design specification (e.g., through backend data storage and syncing).

6. **Visual Hierarchy:**  The visual hierarchy is established through consistent typography and spacing.  The title of each notification type uses a slightly larger font size than the associated toggle, creating a clear relationship.

7. **Variant Selection Rationale:** For the `10:10214` (List item) component, I selected the "Leading: Switch" and "Trailing: None" variants as there is no need for trailing icon.  The switch itself provides sufficient visual feedback for its state. The `10:9912` (switch) component uses the "State: enabled" variant for all switches in the default state.

8. **Native Element Justification:** No native elements were required; all functionality could be implemented using components from the design system.
