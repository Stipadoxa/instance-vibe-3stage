**DESIGN RATIONALE:**

1. **Component Inventory Verification:**  All component IDs referenced in the design specification below are verified to exist in the provided `DESIGN_SYSTEM_DATA`.

2. **Layout Decisions:** The layout uses a vertical layout for the settings page.  Each notification setting is presented as a list item (`10:10214`) with a leading icon representing the notification type, a title describing the notification type, and a trailing toggle switch (`10:9912`) to enable/disable it.  A consistent 16pt spacing is used between list items to improve readability. The horizontalSizing of the text within list items is set to "FILL" so the text fits the width of the item container.

3. **User Flow Reasoning:** The user can easily scan the list of notification types and toggle each one on or off using the switch.  The descriptive title ensures clarity.  The layout prioritizes vertical flow for ease of navigation and comprehension.


4. **Component Choices:** I chose the `10:10214` list item component for displaying each notification setting because it effectively combines a leading element (for the icon), a title (for the description), and a trailing element (for the toggle switch), providing a concise and familiar user interface pattern. It is preferred over using only `native-rows` due to its superior styling and accessibility features and over using a simpler `native-row` due to its limited styling and layout capabilities.


5. **Component Gaps:** The design system lacks a dedicated component that combines a label and toggle switch in a single unit.  This functionality is crucial for the notification settings and therefore I combined the `10:10214` List item with a leading `native-text` and trailing `10:9912` switch. This workaround delivers a visually cohesive and fully functional notification settings interface.

6. **Visual Hierarchy:** Visual hierarchy is established through the consistent use of the `10:10214` list item component.  The placement of the toggle switch is also highly visible, ensuring the primary action is clearly communicated and easily accessible.

7. **Variant Selection Rationale:** For the `10:9912` switch, the `State` variant is set to `enabled` and the `Selected` variant dynamically updates based on user interaction. The `Icon` variant is set to "false" as per the requirements. No other options were relevant. For the list item (`10:10214`), the `Leading` variant is set to "Icon" to clearly represent the notification type. The `Trailing` variant is set to "Switch".  The `Show overline` and `Show supporting text` variants are set to "False".

8. **Native Element Justification:**  Native text elements (`native-text`) are used to display the notification type labels because the system doesn't offer a single component that fulfills that need. Native text allows the customization of font size and style for improved presentation.
