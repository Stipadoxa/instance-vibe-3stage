**DESIGN RATIONALE:**

1. **Component Inventory Verification:** All component IDs used in this design have been manually verified against the provided `DESIGN_SYSTEM_DATA`.

2. **Layout Decisions:** The screen uses a vertical layout to present notification types sequentially.  Each notification type is a list item, consisting of descriptive text and a toggle switch.  The consistent 16px padding on the content section ensures sufficient spacing around elements and provides visual breathing room.

3. **User Flow Reasoning:** The user can easily scan the list of notification types and quickly toggle each option on or off using the switch components.  This simple, direct interaction aligns with the user goal of managing notification preferences efficiently.

4. **Component Choices:** I chose the `10:9912` (switch) component for its clear visual indication of on/off states and its familiarity to users.  Alternatives like custom native shapes would lack the established usability and visual consistency of a dedicated switch component.

5. **Component Gaps:**  The design system lacks a pre-built "list item" component that directly supports a leading icon (notification type icon) and a trailing toggle switch.  I addressed this by using a `layoutContainer` with `layoutMode: "HORIZONTAL"`  to arrange the icon (using `635:4324`  or `635:4329` depending on the toggle state), descriptive text (`native-text`), and `10:9912` (switch) components for each notification type.  This combines existing components to provide the required functionality.

6. **Visual Hierarchy:** The visual hierarchy is established through the use of consistent text styles (primary text for notification type, secondary for brief description) and spacing.  The toggle switches provide a strong visual cue for interaction.

7. **Variant Selection Rationale:** For the switch component (`10:9912`), the "enabled" state is consistently used, adapting the "Selected" variant ("true" or "false") based on the user's preference.  This ensures the switch remains interactive while reflecting the current setting.

8. **Native Element Justification:**  No native elements beyond those explicitly permitted were used. The structure is built exclusively using verified design system components and standard layout containers.
