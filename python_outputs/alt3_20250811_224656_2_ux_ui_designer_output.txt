**DESIGN RATIONALE:**

1. **Component Verification:** All component IDs (`10:5620`, `10:3907`, `10:7773`, `635:4733`) are verified against the provided `DESIGN_SYSTEM_DATA`.

2. **Layout Strategy:** A mobile-first approach is used, starting with a 375px viewport. The screen is divided into sections (Header, Content, potentially Footer if needed for desktop).  The vertical layout is chosen for the main content to maintain a natural reading flow for product details.  The image carousel uses a horizontal wrap layout for efficient space usage.  The action buttons are in a horizontal row for better visual separation and ease of use.

3. **User Experience Logic:** The primary interaction flows are focused on reviewing product details and initiating contact with the seller.  Secondary actions are provided for additional information or reporting.

4. **Component Selection:**  The `appbar` component is used for the header to provide a consistent header structure.  The `Button` component is used for the primary actions for consistency with the design system. The `avatar` component displays the seller's profile picture.  A native `native-text` is used for various text elements, including price, condition, description and specifications due to lack of component equivalents to satisfy the styling requirements.

5. **Gap Analysis:** The design system does not include components for image carousels, hence native rectangles with image fills are used as placeholders.

6. **Hierarchy Decisions:**  The product title in the appbar holds primary visual importance. The price and condition information are emphasized next, followed by the product description and specifications. Visual hierarchy is achieved through font sizes and weights and the use of dividers.  VERTICAL layout is chosen for primary-descriptive content relationships (image, title, description, specifications).

7. **Container Sizing:**  The root container uses `FIXED` sizing modes for mobile viewport simulation, while the content containers use `AUTO` height and `FILL` width to adapt to content and screen. This makes it responsive on various screen sizes.  Horizontal rows use `HUG` horizontalSizing for optimal efficiency.

8. **Variant Rationale:** The `appbar` uses default variants except for the headline and subheadline text. The `Button` components use variants for different types (filled/outlined) to signal different functions (contacting/watchlisting).  The `avatar` component uses photo variant.

9. **Native Element Justification:**  `native-rectangle` elements are used for image placeholders because the design system does not have a suitable carousel component and building one from scratch would exceed project scope.
