Domain & Context
Domain: Account settings management within a mobile application context (Cars45 auction platform implied).
User Type: Registered users of the Cars45 mobile app who need to manage their account settings, personal information, and preferences.
Platform Impact: Mobile users require a streamlined settings experience with clear navigation, concise information, and touch-friendly controls.  Screen real estate is limited.

User Requirements
Primary Goals: Modify personal information (phone, email), manage notification preferences, adjust account settings (language, chat).
Secondary Goals: Access account details (business information, verified ID status), securely change password, delete account, log out.
User Constraints: Limited screen space, touch interaction primary, potential for interruptions, need for clear and concise information.
Ready for handoff if:  All settings options from the reference image are included, navigation is intuitive and consistent with app standards, and all interactive elements have sufficient tap targets.

Content Requirements
Critical:  Personal details (editable), business details (viewable), access to Cars45 auction (toggle), "Verified ID" badge (viewable with explanation), Change phone number (editable), Change email (editable), Change language (selection), Disable chats (toggle), Disable Feedback (toggle), Manage notifications (link to a dedicated screen), Change password (link to a dedicated screen), Delete account (confirmation required), Log out.
Supporting:  Clear visual indicators of active/inactive states (e.g., toggles), informative labels for each setting.
Contextual:  "What is it?" help information for relevant settings.  Confirmation dialogs for account deletion and possibly password changes.

Structural & Component Directives
*   Layout:  Vertical list of settings options with section headers.  Consistent with the reference image.  Top and bottom app navigation bars are to be used.
*   Components:  Use existing app components for list items, toggles, and other interactive elements.  
*   Styling:  Follow existing app style guide.


Screen Requirements
Primary Actions:  Tap on a setting option to modify it (e.g., editing profile information, toggling settings).
Secondary Actions:  Accessing help information (e.g., "What is it?"), navigating to related screens (e.g., password change, notification management).
Screen States:  Default state (all settings displayed), loading state (while saving changes), success state (after successful updates), error states (if updates fail).
Feedback Needs:  Clear visual indicators of success or failure when making changes, clear messaging and user guidance.
